# Career RPG 專案開發進度與架構總結 (Developer Handoff Note)

這份文件為 Career RPG 專案的完整架構、邏輯流程、AI 語意分析校正與標籤系統定義之最新狀態總結。旨在提供後續 AI 助理或開發人員一份清晰、具體且開箱即用的上下文（Context），能立刻掌握專案目前所有細節並接續開發。

## 1. 專案概述與定位 (Project Overview)
*   **專案名稱**：Career RPG (職涯模擬系統)
*   **核心目標**：透過 RPG 的探索與對答形式，引導使用者將模糊的職涯經驗與感受轉化為結構化的職涯向度分析。
*   **分析框架**：系統將流程拆分為 **Want（想不想）** 與 **Able/Can（能不能/可以）** 雙軌，最終透過 AI 語意分析自動對應到 A、B、C、D 四大核心向度的系統化編碼標籤，並於職涯圖鑑（`subpages.html`）中解鎖或對照對應的角色概覽與技能樹。

## 3. 前端流程與路由邏輯 (Frontend Flow & State Management)

前端主要由 `career-flow.html` 與 `subpages.html` 組成，核心邏輯設計如下：

### 3.1 視覺轉場與防誤觸機制
*   **高斯模糊轉場 (Gaussian Blur Transition)**：頁面切換全面棄用滑動效果，改採柔和的 `opacity` + `filter: blur` 的高斯模糊淡入淡出，提升沉浸感。
*   **防誤觸 (Click Prevention)**：按鈕 Keyframes 動畫執行期間設定 `pointer-events: none`，動畫完全呈現後轉為 `auto`，防止使用者在動畫未完成時重複點擊造成非預期路由錯誤。
*   **重整閃爍優化**：在 `window.onload` 初始化載入時，呼叫 `goTo(target, true)`（`immediate = true`）略過退場動畫，直接將當前 `localStorage` 記錄的頁面渲染出來，解決畫面閃爍問題。

### 3.2 狀態持久化與重導向
*   **持久化機制**：使用 `localStorage` 儲存使用者的答題狀態、當前頁面 `pageId`，以及是否已完成第一次探索的解鎖標記 `careerRPG_hasCompletedFirstRun`。
*   **分流頁面 (page-balance)**：在分流頁 `chooseDream()`（別名為「買東西..」頁面）中，系統會依據是否已解鎖/抵達過成果頁進行分流：
    *   已完成探索者：直接引導至歷史回顧頁。
    *   未完成探索者：引導至各自的故事引導流程。
*   **安全重導向**：若使用者未解鎖成果頁卻嘗試直接載入回顧頁，系統會自動將其重導向至引言頁面 `page-dreams-intro`，保護流程的完整性。
*   **路由重設**：支援 URL 參數 `?step=start`，當偵測到此參數時，會自動清空 `localStorage` 中關於 RPG 流程的紀錄，並將路由重新定向至初始引導。

### 3.3 雙軌與雙向綁定 (Two-way Binding)
*   **Want（想不想）軌道**：
    *   新增專屬頁面：`page-event-summary-want` (事件簡述 1)、`page-adjust-want` (調整向度/增加細節)、`page-event-summary-2-want` (事件簡述 2) 及過渡緩衝頁面 `page-semantic-buffer-want`。
    *   **Q1~Q3 雙向綁定**：將「調整向度」頁面替換為三個獨立的 Q1~Q3 輸入框。透過 Javascript 與最初問卷資料連動，修改時會自動同步舊資料，事件簡述也會即時抓取最新問卷結果。
*   **Able/Can（能不能）軌道**：
    *   包含事件描述頁 `page-can-or-not` 與返回邏輯。
    *   返回鍵（goto）會正確指向「買東西..」分流頁面。

## 4. AI 語意分析與判定校正系統 (Semantic Analysis & Calibration)

系統提供**本地/雲端雙軌 AI 分析路由**，當檢測到網址主機名為 `localhost` 或 `127.0.0.1` 時，將自動導向本地 Ollama，否則透過 Cloudflare Pages Functions 調用雲端 API。

```
                              [ 前端 career-flow.html ]
                                       |
                     +-----------------+-----------------+
                     | (window.location.hostname)        |
                     v                                   v
             [ 本地: Localhost ]                 [ 雲端: 生產環境 ]
                     |                                   |
           (POST to Local Ollama)               (POST to /analyze)
      http://localhost:11434/api/generate                |
                     |                                   |
         模型: [career-analyzer]                         v
                                             [ functions/analyze.js ]
                                                         |
                                        +----------------+----------------+
                                        | (USE_CLOUDFLARE_AI === "true") |
                                        v                                 v
                              [ CF Workers AI ]                  [ Groq API Fallback ]
                              @cf/qwen/qwen2.5-                  qwen-2.5-32b
                              coder-32b-instruct
```

### 4.1 5 大否定約束規則 (Exclusion Rules)
為了防範 Qwen 2.5 在零散描述中容易因為關鍵字（如「寫程式、自學、安撫」）而產生誤判，在本地 Modelfile 與雲端 API 中皆強制寫入 5 大否定約束：
1.  **自學與技術研發排除**：禁止將「自學技術、自學 Web 框架、寫程式、研發核心架構」等操作判定為 `D03` 訪談引導。這些應歸為 `D09` 通用數位工具操作，若涉及學習沉沒成本則可判定為 `C14` 通用技術高沉沒成本。
2.  **心靈情緒支持排除**：禁止將「心靈支持」、「安撫家屬情緒」、「安撫焦慮」等精神消耗判定為 `B10` 持續性體能動員 或 `D12` 大肌肉動作執行。精神承接應判為 `B01` 高強度情緒勞動涉入，若有口頭安慰則搭配 `D01` 口語表達。
3.  **資源與目標協調判定**：禁止將「招募志工、分配清潔區域」判定為 `B02` 高密度階層對接 或 `D03` 訪談引導。此為目標協調，應判為 `A10` 多方可接受交集的判斷。
4.  **數據/資訊整理與規律辨識區隔**：禁止將單純用 Excel 或工具整理資料的動作判為 `A03` 大量資訊中的規律辨識 或 `A04` 分析框架的建立與定義。`A03` 必須有明確「看出底層重複規律或痛點結論」；`A04` 必須有「建立可重複操作結構」的分析架構。
5.  **客訴環境判定**：尖峰時刻客訴不斷、場面吵雜，應判為 `B01` 高強度情緒勞動涉入 與 `B06` 高密度感官刺激的工作環境，非 `B02` 或 `A10`。

### 4.2 10 組 Few-Shot 訓練範例
以下 10 組黃金 Few-Shot 範例被同時編入地端 Modelfile 與雲端 `analyze.js` 中，確保了極高的上下文對齊效果：
1.  **程式自學**：自學新 Web 框架、克服瓶頸 -> `B04: 高精準度要求與決策責任承擔`, `C14: 領域專有技術的高沉沒成本`。
2.  **情緒支持**：安撫焦慮家屬、高強度心靈支持 -> `B01: 高強度情緒勞動涉入`, `D01: 口語表達`。
3.  **效率流程**：獨立完成報表系統、精簡流程 -> `A08: 工作流程的診斷與改善判斷`, `D09: 通用數位工具操作`。
4.  **自學工具**：免代碼工具做行銷漏斗、無報酬但有價值 -> `D09: 通用數位工具操作`, `A13: 資源條件與投入時機的判斷`。
5.  **紙本數位化**：十年紙本檔案轉化為高效檢索系統 -> `A08: 工作流程的診斷與改善判斷`, `D08: 數據讀取與分析操作`。
6.  **資料分類**：規劃資料庫架構、克服格式不一、三秒檢索 -> `A03: 大量資訊中的規律辨識`, `D09: 通用數位工具操作`。
7.  **應變指揮**：戶外活動因雨換場、兩小時內指揮搬運 -> `A11: 複合變數失控下的應變判斷`, `A13: 資源條件與投入時機的判斷`。
8.  **客訴現場**：捷運站旁餐廳店長、人手不足客訴不斷 -> `B01: 高強度情緒勞動涉入`, `B06: 高密度感官刺激的工作環境`。
9.  **志工招募**：主動發起清潔日、招募分配區域 -> `A10: 多方可接受交集的判斷`, `D11: 實體空間操作與設備使用`。
10. **駭客松**：極短時間交付具備雲端同步產品原型 -> `A02: 模糊情境下的問題核心辨識`, `D09: 通用數位工具操作`。

### 4.3 判定校正成效與測試結果
*   **測試來源**：經由地端 Ollama 特化模型 `career-analyzer` 跑測 `test_cases.txt` 中 15 個嚴格的 RPG 經歷測試案例。
*   **成效數據**：在未導入約束與 Few-shot 前，模型判定 Exact Match（與預期完全一致）率僅 **6.7%**。在校正並導入 10 組 Few-shot 後，**Exact Match 精準配對率爆炸性暴增至 73.3%**，而平均精準度（Precision）與召回率（Recall）雙雙達到 **83.3%**，成功解決所有嚴重誤判。

## 5. 最新版職涯標籤系統對照表 (The Tag System)

此處整理自 [.agents/skills/teeth-tag/tag-system.md](file:///Users/hongpeiyuan/Desktop/Brian/Programming%20Stuff/antiproject/CareerMatch/.agents/skills/teeth-tag/tag-system.md)，為前端渲染與 AI 辨識之唯一基準。

### ── 向度一：核心職能 (Core Competencies) ──
個人在工作中反覆展現、具有跨情境遷移性的底層能力模式。標籤描述的是認知與策略層的能力模式。
*   **A01　專業內容的跨背景轉譯**：重新組織資訊的層次與語言，使門檻高的專業內容能被不同背景的對方接收。
*   **A02　模糊情境下的問題核心辨識**：從不完整的資訊中，判斷出「真正的問題是什麼」與「範圍界線」。
*   **A03　大量資訊中的規律辨識**：面對大量零散的資訊，辨識出底層規律、重複訊號或連動關係。
*   **A04　分析框架的建立與定義**：將規律或判斷結果，組織為一個明確維度與邊界、可重複操作的框架。
*   **A05　概念本質的釐清與定位**：判斷一個尚未成形的概念其核心主張、邊界與要解決的本質問題。
*   **A06　概念到執行方向的結構拆解**：將概念或想法拆解為具體可推進的方向、步驟與優先順序。
*   **A07　長程目標的規劃與查核設計**：評估長程目標資源與條件缺口，拆解為里程碑並設計完成標準與查核點。
*   **A08　工作流程的診斷與改善判斷**：觀察現有工作模式，辨識冗餘與失誤風險，並設計更穩定的流程結構。
*   **A09　多方立場與底線的解讀**：跨越多次互動，判斷存在利益或立場差異的多方真正在意什麼與退讓空間。
*   **A10　多方可接受交集的判斷**：在掌握多方立場後，整合判斷出能被各方接受、折衷可行的平衡方案。
*   **A11　複合變數失控下的應變判斷**：在多個條件（時間、資源、變數）同時出錯下，重新評估並做出結構性決策。
*   **A12　實體空間使用邏輯的規劃判斷**：針對使用需求，規劃實體空間的配置、動線與物件擺放邏輯。
*   **A13　資源條件與投入時機的判斷**：在資訊不完整下，評估資源（時間/能力/成本）並決定此時是否值得投入。
*   **A14　特定對象行為模式的預測判斷**：根據跨時間的累積觀察，推導與預判特定族群在特定情境的下一步反應。
*   **A15　規則與情境的適用條件判斷**：面對既有規則，判斷與詮釋其在特定情境下的適用度與解釋邊界。

### ── 向度二：環境負荷 (Environmental Load) ──
工作情境對個人在心理、生理、認知層面持續施加的結構性壓力。標籤描述的是一個人需要長期承受的外部工作型態與需求。
*   **B01　高強度情緒勞動涉入**：長期在職業要求下承接他人的強烈情緒（憤怒/悲傷/恐懼/崩潰），並維持專業姿態。
*   **B02　高密度階層對接之認知負荷**：短時間內反覆切換對接不同權力位置的對象，需快速調整溝通語言與應對框架。
*   **B03　高頻目標調整之持續應對需求**：計畫修正是結構性常態，需在舊方向未完下重新定向，面臨調適要求。
*   **B04　高精準度要求與決策責任承擔**：錯誤容許空間結構性接近於零，需長期維持高正確率並承擔決策後果。
*   **B05　多方利益平衡之持續角色要求**：角色結構要求同時回應利益或期待存在根本性張力的多方，需維持平衡。
*   **B06　高密度感官刺激的工作環境**：工作場域長期處於高音量、視覺資訊密集、持續被打斷等高強度感官輸入狀態。
*   **B07　低外部刺激與高重複性的工作型態**：內容長期高度重複、低變化，缺乏外部刺激、即時回饋與新鮮感。
*   **B08　低外部節律依賴的獨立作業型態**：缺乏團隊節奏或明確里程碑，需在缺少外部節律錨點下長期管理自主作業。
*   **B09　工作與生活時間的高度交疊性**：工作需求持續滲入非工作時間，難以在生活建立穩定的恢復空間。
*   **B10　持續性體能動員的工作型態**：每天持續動用大肌群或要求長時間維持體能輸出，身體承受物理負荷。
*   **B11　高強度他人困境的持續涉入**：持續接觸他人的創傷經歷、極端困境或生命痛苦，且被要求維持專業狀態。
*   **B12　持續性公眾可見度與表現要求**：言行、形象或表現長期處於公眾或評價性場域的觀察與評價之下。
*   **B13　多工並行與高資訊密度的工作型態**：結構性要求同時維持多條並行任務線，處理隨時湧入的高密度資訊。
*   **B14　低結構化指令與自主定義標準的工作型態**：缺乏明確任務指令和績效標準，必須自主定義「做到什麼程度算完成」。
*   **B15　非常規作息與生理節律的持續調適需求**：排班性質（輪班/夜班）持續干擾生理作息，無法建立穩定睡眠週期。
*   **B16　特定物理環境條件的持續工作型態**：場域物理條件（高低溫/粉塵/化學/噪音）對身體健康有累積性影響。

### ── 向度三：現實門檻 (Entry & Sustain Conditions) ──
進入或持續留在某個職涯領域所需承擔的結構性成本與限制條件。
*   **C01　法定執業資格門檻**：執業前強制要求通過國家考試或取得法定執照，與個人能力無關。
*   **C02　學歷背景的結構性篩選**：業界慣例要求特定學位或學校背景作為入場篩選前提。
*   **C03　強制過渡期的財務壓縮結構**：制度規定且無法縮短的低薪實習、見習或助理期，時間與財務受到雙重壓縮。
*   **C04　非公開管道的機會依賴結構**：機會與案源高度依賴封閉圈子或內部引薦，外來者缺乏公開競爭管道。
*   **C05　作品集與可見成果的入場前提**：入場前需要備有一定規模的作品集或發表記錄，此累積期通常缺乏穩定回報。
*   **C06　財務回報的結構性延遲**：入行到獲得穩定財務回報需要跨越較長的時間窗口，需要前期的財務緩衝能力。
*   **C07　收入波動與現金流不確定性**：薪酬以案計酬或與業績高度連動，在時間維度上存在劇烈波動。
*   **C08　入行初期的財務資本墊付**：入行初期需自行負擔高昂設備、進修、認證或場地成本，或面臨高設備維護成本。
*   **C09　薪資結構的制度性天花板**：薪資增長依賴年資或職級，而非個人表现，上限在職涯前期即可預期。
*   **C10　非常規排班的強制接受**：必須接受輪班、夜班或值班制的強制排班，無法選擇固定班表。
*   **C11　高頻率地理移動的工作型態**：工作執行本質要求常態性跨區移動或出差，地理位置不固定。
*   **C12　執業場域的地理綁定限制**：工作無法遠端且必須在特定地點（或因制度分發）執業，移動彈性極低。
*   **C13　職涯機會的地理集中性**：核心機會高度集中於特定城市，不在該地區者面臨結構性的機會劣勢。
*   **C14　領域專有技術的高沉沒成本**：所學工具或技術通用性極低，一旦離開此領域或企業，其價值難以轉移。
*   **C15　技術迭代造成的持續再投入需求**：核心技術迭代極快，必須持續投入時間與財務更新技能以防貶值。
*   **C16　職涯出口的結構性窄化**：轉職路徑窄、可選出口少，跨領域轉換面臨極高摩擦成本。

### ── 向度四：可執行技能 (Executable Skills) ──
個人能夠直接調用、用以完成具體任務的操作性技能。描述的是主動運用某項能力達成任務的表現，排除純粹的思考或判斷過程。
*   **D01　口語表達**：口頭語音傳遞資訊、回應問題、推進對話（在場互動或準備好架構）。
*   **D02　主動傾聽與語義提取**：口頭溝通中辨識超越字面的真實意圖、情緒與核心需求，並據此回應。
*   **D03　訪談與提問引導**：帶明確目的設計問題，引導對方開口並即時調整問法（非協調任務或安撫情緒）。
*   **D04　肢體示範與動作引導**：以身體動作為媒介，向他人示範操作、引導移動路徑或傳遞指令。
*   **D05　結構化書面輸出**：整理為邏輯清楚、層次分明的文字內容（如報告、大綱），讓讀者能使用。
*   **D06　視覺化資訊呈現**：數據、流程、概念轉化為圖表或視覺形式，或解讀他人的視覺資訊。
*   **D07　質性資訊蒐集與整理**：從訪談、觀察、田野、文件中提取、分類與記錄內容（非規律辨識）。
*   **D08　數據讀取與分析操作**：讀懂數字意義並執行基礎至進階統計處理以支援決策。
*   **D09　通用數位工具操作**：操作文書處理、簡報、試算表等跨職涯通用數位工具。
*   **D10　數位溝通工具操作**：操作通訊軟體、視訊會議、協作與專案管理平台維持日常協作。
*   **D11　實體空間操作與設備使用**：在實體環境中操作特定設備、儀器或工具，與空間物件直接互動。
*   **D12　大肌肉動作執行**：動員全身或大型肌群完成體能性任務（如移動物品、體能陪伴）。
*   **D13　小肌肉精細操作**：手眼協調或手部精細控制，完成單次/短週期任務。

## 6. 給後續 AI 開發者的接續指引 (Next Steps)

1.  **成果頁標籤渲染 (Tag Visualization)**：
    *   目前後端語意分析回傳格式為：`{"tags": ["A01: 專業內容的跨背景轉譯", "B03: 高頻目標調整之持續應對需求"]}`。
    *   前端接收到此 JSON 陣列後，應有對應的 Parser 將程式碼（如 `A01`）解鎖或轉換，並在 `page-semantic-analysis-want` 或 `page-semantic-analysis-able` 中以精緻、符合整體質感的 UI（例如微動畫、不同向度顏色的標籤卡片）渲染出標籤的**名稱**與**詳細中文定義**。
2.  **CORS 與 Ollama 開發環境設定**：
    *   若在本地調用 Ollama 服務時遭遇跨來源資源共用 (CORS) 問題，請提示使用者在本地終端機執行下述命令重啟 Ollama：
        ```bash
        launchctl setenv OLLAMA_ORIGINS "*"
        ```
    *   若要重構或更新本地特化模型，可在終端機於專案根目錄下執行：
        ```bash
        ollama create career-analyzer -f CareerAnalyzer.Modelfile
        ```
